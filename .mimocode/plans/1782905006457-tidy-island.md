# Plan: Add Sync + Async Dual Support

## Goal
Auto-detect engine type (`AsyncEngine` vs sync `Engine`) and transparently support both. When async: current behavior unchanged. When sync: use `Session` + `def` handlers (FastAPI runs them in a threadpool).

## Core Principle
When engine is sync, `session.execute()` returns `Result` directly (no await). Therefore **every function in the call chain must be sync**. FastAPI natively supports both `async def` and `def` route handlers, so we register the appropriate type based on engine detection.

## Revised Simplified Plan

### Architecture

```
Async engine path (current):
  Async def handler → await session.execute() → await session.commit()

Sync engine path (new):
  def handler → session.execute() → session.commit()
  (FastAPI runs def handler in threadpool)
  (Dependencies like permissions/CSRF still resolved async by FastAPI)
```

## Risk Mitigation

- **Form parsing in sync handlers**: Use async deps for pre-parsing (FastAPI resolves deps async even for sync routes)
- **Storage operations**: File I/O is inherently sync; wrapping in async was unnecessary overhead in sync mode
- **Testing**: Add dedicated sync test suite alongside existing async tests
- **Backward compatibility**: Async path is untouched; sync is additive only
