# Plan: Add SQLModel Support to fastapi-console

## Goal
Allow users to register SQLModel-based models alongside existing SQLAlchemy DeclarativeBase models in the admin panel.

## Key Insight
SQLModel inherits from SQLAlchemy's `DeclarativeBase`, so `sqlalchemy.inspect()` works natively. Most of the existing code (inspection, widgets, views, forms) already works with SQLModel models. Only **3 files** need real changes.

## Changes

### 1. NEW: `fastapi_console/compat.py`
Detection helpers for SQLModel:
- `is_sqlmodel_class(model)` — `isinstance` check against `sqlmodel.SQLModel`
- `is_sqlmodel_table(model)` — has `__tablename__` (i.e., `table=True`)
- Guard with try/except ImportError since sqlmodel is optional

### 2. MODIFY: `fastapi_console/registry/core.py`
Update `auto_discover()` (line 219) to also scan `SQLModel.__subclasses__()` after the existing DeclarativeBase walk. SQLModel models are already in the DeclarativeBase tree, but the second pass handles edge cases with multiple inheritance.

Also update docstring to mention SQLModel support.

### 3. MODIFY: `fastapi_console/registry/validation.py`
In `_validate_is_sqlalchemy_model()` (line 44): when `__tablename__` is missing, check if it's a SQLModel with `table=False` and raise a clearer error message pointing users to use `table=True`.

### 4. MODIFY: `pyproject.toml`
Add optional dependency: `sqlmodel = ["sqlmodel>=0.0.14"]`

### 5. NEW: `tests/test_sqlmodel_support.py`
Tests guarded with `@pytest.mark.skipif` when SQLModel is not installed:
- Detection helpers work correctly
- SQLModel `table=True` models register successfully
- SQLModel `table=False` schemas are rejected with clear error
- Auto-discovery finds SQLModel models
- Column/relationship metadata extraction works on SQLModel models

## Files NOT modified
- `inspection/registry.py` — `sqlalchemy.inspect()` works on SQLModel natively
- `widgets/registry.py` — maps SQLAlchemy types, unaffected
- `views/class_views.py` — CRUD uses session APIs, unaffected
- `db.py` — AsyncSession/async_sessionmaker work with any SA engine
- `models/base.py` — admin-internal models stay pure SQLAlchemy
