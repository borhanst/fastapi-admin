# Plan: Clean Single-Role Permission System

## Problem

The user form has two confusing role fields ("Primary Role (Legacy)" dropdown + "Roles (Multi-Select)"). The user wants **one clean role field** and the permission system to work correctly.

## Changes

### 1. `fastapi_console/templates/pages/users/form.html` — Single role field

**Current**: Two fields: legacy `role_id` dropdown + `role_ids` multi-select.
**New**: Remove the multi-select. Keep only the single role dropdown. Rename label from "Primary Role (Legacy)" to "Role". Update help text.

### 2. `fastapi_console/auth/permissions.py` — Fix PermissionChecker

**Current**: Only queries single `role_id` — this is correct for single-role model, but the logic works fine as-is.
**Keep as-is** since we're using single role assignment.

### 3. `fastapi_console/views/users.py` — Simplify user management

**Current**: Handles both `role_id` and `role_ids` from form.
**New**: Only handle `role_id` from form. Remove multi-role logic from create/edit handlers.

### 4. `fastapi_console/templates/pages/users/form.html` — Add permission summary

Add a read-only section below the role dropdown showing what permissions the user will have from their selected role (table-level view/create/edit/delete). This gives admins immediate visibility into what access a user gets.

### 5. `fastapi_console/templates/pages/users/list.html` — Show role name

Add a "Role" column to the user list table showing the assigned role name.

### 6. `fastapi_console/admin/builtin_models.py` — Update list_display

Add `role` to `AdminUserAdmin.list_display` so role is visible in the admin grid view.
